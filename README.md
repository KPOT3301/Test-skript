1. Инициализация и подготовка
Устанавливается уровень логирования, часовой пояс (для метки времени), глобальные счётчики.

Загружается база GeoIP (GeoLite2-City.mmdb) – если её нет, она скачивается с GitHub.

Определяются списки User-Agent для случайного выбора при HTTP-запросах.

2. Чтение источников
Из файла sources.txt читаются строки (каждая может быть прямой ссылкой на ключ или URL подписки).

Для каждого источника:

Если это уже готовая ссылка (vless://, ss:// и т.д.) – добавляется в общий набор.

Иначе – выполняется HTTP-запрос, полученный контент декодируется из Base64 (если это подписка), из текста извлекаются все ссылки регулярным выражением.

Подсчитывается статистика: сколько ссылок получено из каждого источника, общее количество уникальных ссылок.

3. Дедупликация по IP (до TCP-проверки)
Для каждой ссылки быстро (без TCP-соединения) определяется IP-адрес домена через DNS. Используется поддержка IPv6 (если включено), приоритет можно настроить.

Оставляется только одна ссылка на каждый уникальный IP (берётся первая попавшаяся). Это сокращает количество проверок для серверов, которые используют один и тот же IP для разных протоколов/портов.

Логируется: сколько IP удалось разрешить, сколько уникальных IP осталось.

4. Этап 1: TCP-проверка
Запускается пул потоков с динамически регулируемым числом воркеров (не больше заданного максимума и не больше количества ссылок).

Для каждой ссылки:

Парсится, извлекаются хост, порт, протокол и параметр explicit_sni.

Shadowsocks не требует SNI, поэтому проверка наличия SNI для него пропускается.

Для остальных протоколов: если нет явного SNI – ссылка отбрасывается.

Выполняется несколько попыток TCP-соединения к IP:порт, замеряется задержка. Если хотя бы одна попытка успешна и задержка меньше MAX_LATENCY_MS – ссылка проходит.

Результат: список ссылок, успешно прошедших TCP, с их IP и минимальной задержкой.

5. Определение геоданных
Для каждого IP из успешных TCP-соединений через GeoIP определяются страна, город и флаг (эмодзи).

Даже если страна не определена, ссылка не отбрасывается (флаг и код страны остаются пустыми) – она будет участвовать в следующем этапе.

6. Этап 2: Реальная проверка через Xray
Снова пул потоков с динамической регулировкой (максимум – REAL_CHECK_MAX_WORKERS).

Для каждой ссылки:

Запуск Xray: создаётся временный JSON-конфиг, процесс Xray стартует, открывается SOCKS5-прокси на случайном локальном порту.

Через прокси выполняются несколько проверок последовательно:

HTTP – запрос к тестовым URL (например, http://connectivitycheck.gstatic.com/generate_204). Фиксируется задержка, если она превышает лимит – проверка провалена.
HTTPS – если протокол требует TLS (определяется по параметрам tls, security и т.д.), выполняются аналогичные запросы к HTTPS-адресам.
DNS-over-HTTPS – запрос к 1.1.1.1/dns-query для проверки резолвинга через прокси.
Джиттер (вариация задержки) – серия HEAD-запросов к заданному хосту, вычисляется среднее абсолютное отклонение; если оно превышает MAX_JITTER_MS – сервер считается нестабильным и отбрасывается.
Проверка обхода блокировок (DPI) – выполняются запросы к списку сайтов, которые обычно заблокированы (например, rutracker.org, instagram.com). Если настроено DPI_REQUIRE_ALL = False, достаточно одного успешного доступа; иначе нужны все.
Если все проверки пройдены в пределах таймаутов и лимитов – ссылка помечается как рабочая.

Процесс Xray завершается, временный конфиг удаляется.

Логируется прогресс: для каждой ссылки выводится протокол, результат (✅/❌), сокращённый адрес и TCP-задержка.

7. Формирование итоговых списков
Все рабочие ссылки с их геоданными (флаг, город, страна, задержка) собираются в один список.

Разделяются на две группы:

Российские (country_code == 'RU')

Остальные (включая те, у которых страна не определена)
